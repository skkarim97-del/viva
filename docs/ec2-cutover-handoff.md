Generated by the pre-pilot QA pass on May 03, 2026. This is the

## What this commit changed in source

### Tier A — viva-dashboard (live the next time the dashboard artifact is deployed)

`artifacts/viva-dashboard/.replit-artifact/artifact.toml` line 33:

```diff
- API_ORIGIN = "https://viva-ai.replit.app"
+ API_ORIGIN = "https://api.itsviva.com"
```

This is the dashboard's server-side reverse proxy target. The
dashboard URL (the host the doctor types into the browser) does not
change — only its `server.mjs` upstream changes. Browser-side code
keeps using same-origin `/api/...` calls, so no CORS work is required
on the EC2 box for the dashboard origin.

**Effect:** the next prod redeploy of the viva-dashboard artifact will
proxy `/api/*` to `https://api.itsviva.com` instead of
`https://viva-ai.replit.app`.

**Rollback:** revert this one line and redeploy. ~60 seconds.

**Dev preview is unaffected:** `services.development.run` runs the
Vite dev server and bypasses `server.mjs` entirely, so local preview
keeps proxying to the local API.

### Tier C — pulse-pilot (only affects future EAS builds)

`artifacts/pulse-pilot/lib/apiConfig.ts`:

```diff
- const PRODUCTION_API_URL = "https://viva-ai.replit.app/api";
+ const PRODUCTION_API_URL = "https://api.itsviva.com/api";
```

`artifacts/pulse-pilot/app.json`:

- `ios.buildNumber`: `42` → `43`
- `ios.associatedDomains`: now lists **both** `api.itsviva.com` AND
  `viva-ai.replit.app` so old TestFlight builds don't lose universal
  links during the bake-in window.
- `android.versionCode`: `2` → `3`
- `android.intentFilters[0].data`: now lists **both** hosts for
  the same reason.

**Effect:** the next EAS build picks these up. **Already-installed
pulse-pilot apps are unchanged** — `PRODUCTION_API_URL` is a bake-time
constant baked into the JS bundle, so the only way a phone in the
field starts hitting `api.itsviva.com` is by installing a build made
from this commit or later.

**Rollback for shipped apps:** there is none — see the runbook's
"Mobile after Phase 12" row of the rollback table. The mitigation is
to keep the Replit autoscale deployment alive and serving the same
RDS for the entire bake-in window so the old cohort keeps working.

### Not changed — viva-analytics

`viva-analytics` is a static SPA (`serve = "static"` in its
`artifact.toml`). It uses same-origin `/api` calls. There is no
`server.mjs` proxy to flip. To point analytics at EC2 you would
need to either:

1. Add CORS allowlist for the analytics origin on the EC2 box AND
   change the analytics fetch helper to use absolute
   `https://api.itsviva.com/api/...` URLs, OR
2. Front analytics with a tiny Node proxy similar to
   `viva-dashboard/server.mjs`.

Neither is needed for the pilot — analytics is internal-only and
keeps reading the same RDS via the existing path. Flipping it is
explicitly out of scope here.

## What is REQUIRED on the EC2 box before universal links work

Universal/App Links currently 404 on EC2:

```bash
$ curl -sS https://api.itsviva.com/.well-known/apple-app-site-association
APPLE_TEAM_ID not configured
$ curl -sS https://api.itsviva.com/.well-known/assetlinks.json
ANDROID_APP_SIGNING_SHA256 not configured
```

This is the route handler at `artifacts/api-server/src/routes/wellknown.ts`
deliberately returning 404 with a placeholder string when the env vars
are not set (better than serving a wrong file Apple/Google will cache).

### To fix on the EC2 box

SSH in and edit `/etc/viva-api.env` (the systemd unit's
`EnvironmentFile`):

```bash
# Apple developer portal -> Membership -> Team ID (10 chars, e.g. ABCD123XYZ).
APPLE_TEAM_ID=<10-char team id>

# Output of `keytool -list -v -keystore <upload-keystore.jks> -alias <alias>`,
# the line `SHA256: AA:BB:CC:...` (64 hex pairs separated by colons,
# uppercase).
ANDROID_APP_SIGNING_SHA256=AA:BB:CC:DD:...
```

Then restart and verify:

```bash
sudo systemctl restart viva-api
curl -sS https://api.itsviva.com/.well-known/apple-app-site-association | head
curl -sS https://api.itsviva.com/.well-known/assetlinks.json | head
```

Both must return JSON (not the placeholder error string). The
expected shapes are documented in `wellknown.ts:24-89`.

### Other env vars that must already be set on EC2

Per `scripts/ec2/viva-api.env.example`, the EC2 service requires:

- `AWS_DATABASE_URL` (so it writes to the same RDS the Replit autoscale
  deployment writes to)
- `SESSION_SECRET`
- `ALLOWED_ORIGINS` — comma-separated, must include
  `https://viva-ai.replit.app` AND the new dashboard origin (whatever
  the published dashboard URL is) so CORS preflights succeed during the
  bake-in window
- `INTERNAL_API_KEY` (for `/api/internal/*` operator endpoints)

The `/api/healthz` 200 response we observed at
`https://api.itsviva.com/api/healthz` proves the systemd unit is up
and the DB connection succeeded — but we did NOT independently
verify on the box that `AWS_DATABASE_URL` is the same RDS instance
this Replit reads. The runbook's smoke step
(`docs/ec2-smoke-test.md` step 1.7) is the way to prove this, and it
should be run with operator access to the box.

## What I deliberately did NOT touch

- Did not run any destructive RDS migration. The schema on RDS already
  matches the source (verified column-by-column for
  `patient_interventions` including `feedback_result`,
  `feedback_collected_at`, `patient_note`).
- Did not point analytics at EC2 (would require CORS rules on EC2 +
  absolute URLs in the analytics fetch helper — out of scope for
  pilot readiness).
- Did not delete the legacy "helium" Replit DB. It is unused
  (`AWS_DATABASE_URL ?? DATABASE_URL` in `lib/db/src/index.ts`
  prefers RDS) but kept as backup per the autonomy rules.
- Did not flip iframes on the canvas board. Those are designer-only
  visual artifacts, not part of the cutover.
- Did not enable `ENABLE_DEV_LOGIN=true` on EC2. EC2 is
  `NODE_ENV=production`, which means `/api/dev/*` returns 404 there
  by design. Demo reset only works against the Replit autoscale
  deployment, which is intentional.
