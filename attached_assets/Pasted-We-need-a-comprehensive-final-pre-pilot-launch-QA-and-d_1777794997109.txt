We need a comprehensive final pre-pilot launch QA and data-flow cleanup for Viva.

This is very important. The goal is to verify that the MVP is truly pilot-ready across:
- Viva Care patient app
- Viva Clinic provider dashboard
- Viva Analytics
- API/backend
- AWS database
- deployment/runtime environment
- data persistence
- intervention intelligence
- escalation workflow
- analytics/event tracking

Do not make assumptions. Audit first, then fix only what is broken or incomplete.

Important:
- Do NOT change users.id from serial
- Do NOT make risky DB changes
- Do NOT break login, invite flow, onboarding, Today tab, Week tab, Trends tab, Viva Clinic, or Viva Analytics
- Do NOT send PHI to OpenAI
- Preserve HIPAA / PHI guardrails
- No fake production behavior except explicit dev/demo seeding
- Demo/seeded data should be clearly dev-only
- Production/pilot patient and provider data must persist to AWS database
- Nothing important should remain only in local storage, except temporary UI cache or demo/dev-only state

==================================================
PRIMARY GOAL
==================================================

Perform a full pre-pilot QA pass to ensure:

1. All patient data is saved to the AWS Postgres database
2. All provider data is saved to the AWS Postgres database
3. All meaningful app events are logged for analytics
4. All onboarding data is saved and used by the intelligence layer
5. All patient symptom inputs are saved as current state and event history

10. Provider escalation works and includes context
11. Today / Week / Plan choices are saved, including automated suggestion vs patient-chosen override

13. Data flows correctly from Viva Care → backend → AWS DB → Viva Clinic → Viva Analytics
14. No important pilot data is only stored locally
15. Runtime/deployment environment is clear and production-safe
