PHASE 1: LOCK DOWN DOCTOR-TO-PATIENT DATA ISOLATION
Audit all clinician-facing routes, queries, loaders, actions, and APIs.

Requirements:
- Every patient must belong to a specific doctor via doctor_id
- Every clinician-facing query must be scoped by the authenticated doctor session
- No doctor should ever be able to access another doctor’s patients by changing URL params, patient IDs, or request payloads
- The backend must not trust raw patient IDs alone for dashboard access

Enforce this rule everywhere:
- patient list
- patient detail
- patient check-ins
- notes
- escalations / follow-ups
- risk summaries
- invite status
- any dashboard metrics

Implementation rule:
Whenever a doctor is requesting patient data, scope access using the authenticated doctor session and patient ownership, effectively:
patients.doctor_id == authenticated_doctor.id

Please:
1. Find every place this is currently enforced
2. Find every place it is missing or weak
3. Fix all of them
4. Add tests for cross-doctor access denial

PHASE 2: CLEAN ONBOARDING AROUND INVITE TOKENS
I want onboarding to be driven by secure doctor-created invite tokens, not by loose matching on phone or email.

Desired flow:
1. Doctor logs into Viva Clinic
2. Doctor clicks Invite Patient
3. Backend creates an invite tied to that doctor
4. Invite generates a secure activation link
5. Patient opens link
6. App or activation page validates token with backend
7. Backend creates or links the patient record to that specific doctor
8. All future patient data is attached to that patient and visible only to that doctor

Please inspect what exists today and refactor toward this model.

Add or clean up an invites table/model with fields like:
- id
- doctor_id
- patient_id nullable until activation
- phone nullable
- email nullable
- token_hash or secure token storage approach
- status: pending / redeemed / expired / revoked
- expires_at
- redeemed_at
- created_at
- updated_at

Requirements:
- invite token must be unique and secure
- token should expire
- token should be revocable
- redeemed tokens should not be endlessly reusable unless explicitly intended
- patient-doctor linking should come from the invite token, not inferred primarily from email/phone

Assume patient distribution may occur through an external TestFlight group. Do not rely on TestFlight identity, App Store Connect tester metadata, or public-link enrollment data to link a patient to a doctor.
