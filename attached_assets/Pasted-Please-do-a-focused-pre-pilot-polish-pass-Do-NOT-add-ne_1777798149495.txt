1. PROVIDER DASHBOARD ESCALATION CARD
==================================================

Goal:
When a patient is escalated, the provider should immediately understand:
- what changed
- why Viva surfaced the patient
- what Viva recommended
- what the patient did
- what the patient reported
- what the provider should review

Current issue:
Escalations should not look like a generic “patient requested review.” They need structured, useful clinical context.

Please audit the Viva Clinic provider dashboard and find where escalated patients / review requests are shown.

Update the escalation card or patient detail escalation section to include:

A. Clear header
Example:
Review requested
or
Patient needs review

B. Priority / reason chip
Examples:
- Nausea worsening
- Severe symptoms
- Symptoms not improving
- Patient requested review
- Reported worse after support
- Around dose timing
- Low intake + nausea

C. Current symptoms
Show structured symptom state:
- Energy: Tired
- Appetite: Low
- Nausea: Severe
- Digestion: Constipated
- Bowel movement: No

D. Recent trend
Show short trend if available:
- Nausea has shown up 2 times this week
- Appetite has been lower across recent check-ins
- Symptoms appear around dose timing
- Activity is lower than usual, if Apple Health data exists
- No trend yet, if not enough data

Do not overclaim causation. Use “shown up,” “appears,” “tends to,” or “recently.”

E. Viva recommendation shown
Show what the patient saw:
- Primary recommendation title
- Recommendation body
- Support categories if available, e.g. hydration, appetite, digestion

Example:
Viva recommended:
“Settle nausea without skipping nutrition”
Try a few bites of a bland, protein-forward snack and sip water slowly.

F. Patient action / feedback
Show:
- Patient tapped “I’ll try this” or not
- Another option tapped, if available
- Feedback: Better / Same / Worse
- Review requested timestamp

Examples:
Patient action:
Tried recommendation · Feedback: Worse

or:
Patient requested review before trying support

G. Suggested provider action
Keep this non-prescriptive:
- Review symptoms and recent support history
- Consider whether outreach is needed
- Review if symptoms are persistent, worsening, or hard to manage

Avoid medication advice or dose-change recommendations.

H. Review workflow
If currently supported:
- Mark reviewed
- Log action taken
- Follow-up sent
- No action needed

If not currently supported, do not build a large workflow. At minimum, make sure the escalation card has enough context and a clear “Reviewed” action if already available.

Data source requirements:
Use existing DB data where available:
- care_events
- patient_checkins
- patient_interventions
- analytics_events
- patient_treatment_logs
- health summaries if available
- patient_plan_items if useful

Do not duplicate data if already in one place.
Do not expose patients outside provider scope.

Acceptance:
- Provider can open Clinic and understand why a patient was escalated in under 10 seconds
- Escalation card shows symptoms, trend, Viva recommendation, patient feedback, timestamp and review reason
- No generic contextless escalation cards remain
- Provider scoping remains intact
- No PHI is sent to OpenAI

4. DEMO MODE RESET
==================================================

Goal:
Add a safe dev-only demo reset so we can reliably show stable, moderate and severe scenarios during demos without manually fighting app state.

Important:
This must be dev-only.
Do not enable in production.
Do not affect real patients.
Use demo doctor / demo patients only.
Do not delete real data.
Do not expose this to production users.

Preferred implementation:
Add a dev-only endpoint and/or dev-only button.

Option A: Backend endpoint
POST /api/dev/reset-demo-patient

Body:
{
  "scenario": "stable" | "moderate" | "severe"
}

Behavior:
- Only enabled in development or ENABLE_DEV_LOGIN=true
- Finds or creates demo patient
- Clears or supersedes active demo-only intervention state as needed
- Seeds today’s check-in for the selected scenario
- Seeds relevant plan items
- Seeds integration state if helpful
- Seeds health data if scenario requires it
- Returns demo patient token or confirms reset

Scenarios:

Stable:
- Energy: Good or Great
- Appetite: Normal or Strong
- Nausea: None
- Digestion: Fine
- Bowel movement: Yes
- Symptom support should be preventive or minimized

Moderate:
- Energy: Tired
- Appetite: Low
- Nausea: Moderate
- Digestion: Constipated
- Bowel movement: No
- Symptom support should show moderate support

Severe:
- Energy: Depleted
- Appetite: Very Low
- Nausea: Severe
- Digestion: Diarrhea or Constipated
- Bowel movement: No
- Symptom support should show heavier support and care-team review option

Option B: Dev-only UI button
On dev login or demo screen, show:
- Reset demo: Stable
- Reset demo: Moderate
- Reset demo: Severe

After tapping:
- loads Today tab
- card reflects selected scenario
- no manual symptom entry required

Data cleanup rules:
- only touch demo patient records
- do not delete shared schema data
- avoid destructive deletes if superseding/expiring old demo records works
- make reset idempotent
- repeated reset should not create duplicate active cards
- demo rows should be clearly attributable to source = demo_seed or dev_reset

Acceptance:
- In Replit preview, I can reset demo patient to Stable, Moderate, or Severe
- Today tab immediately reflects the scenario
- Symptom support and status banner update correctly
- No duplicate cards
- Provider dashboard can show demo escalation scenario if severe review is requested
- Dev reset endpoint returns 404 or disabled behavior in production
